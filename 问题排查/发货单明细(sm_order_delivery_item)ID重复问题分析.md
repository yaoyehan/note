# 程序异常分析

1. ## 分析该异常可能的产生原因。
    

目前订单中心大部分表的主键都采用雪花算法生成，其具体生成格式如下：
![[雪花算法.png]]

该算法的特点是在同一个JVM内无论并发多高，都不会产生重复的ID，而不同JVM内要产生重复ID，必须满足以下三个条件：
![[发货单id重复排查日志1.png]]
1. 相同的**DatacenterId**（相同MAC地址一定相同，不同MAC地址可能相同，取值范围0-31）。
2. 相同的**WorkerId**（相同PID一定相同，不同PID可能相同，取值范围0-31）。
3. 发生在同一毫秒内（不同JVM内并发调用可能相同）。

# 问题总结

通过上面问题的排查可以发现，最近之所以会出现很多ID重复的报错。最关键就是因为business-order-center-man和business-order-center-server两个应用在重启之后分配的**PID会生成相同的WorkerId**，而在下次发版重启后PID会发生改变，到时产生的WorkerId也许又不同了。为了降低这种WorkerId相同的概率，我们可以把DatacenterId

和WorkerId都改为随机生成（重复的概率为1/1024），只需给MyBatis Plus增加以下两项配置：

```xml
mybatis-plus:
  global-config:
    worker-id: ${random.int(32)}
    datacenter-id: ${random.int[32]}
```

虽然上面这种方式并不能完全规避问题，但是由于IdWorker.getId()使用的地方有177处，想要从根本上解决ID重复，改造成本非常大，也容易导致新的问题。因此可以考虑在后续订单中心重构的时候，选择更可靠的ID生成方式。
